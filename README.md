# AI Lead Extractor Bot

Telegram-бот, которому пересылаешь сырое сообщение клиента, а он возвращает
аккуратную карточку лида (кто клиент, контакт, что хочет, срочность).
Внутри — бесплатная модель OpenRouter.

**Стек:** Elixir/OTP (конкурентность, супервизия, устойчивость) + небольшой
Rust-NIF (нормализация текста и дедуп-отпечаток входящих сообщений).

## Требования

- Elixir `~> 1.19` / Erlang OTP 29
- Rust `~> 1.92` (`cargo` в PATH — нужен для сборки NIF)

## Настройка

```sh
cp .env.example .env
# заполни TELEGRAM_BOT_TOKEN (от @BotFather) и OPENROUTER_API_KEY
mix deps.get
mix compile
```

## Запуск

```sh
iex -S mix        # интерактивно
# или
mix run --no-halt # как сервис
```

Без `TELEGRAM_BOT_TOKEN` приложение стартует, но бота не поднимает (пишет
предупреждение в лог) — удобно для локальной проверки без токена.

## Линтеры и анализ

```sh
mix format                 # форматирование
mix credo --strict         # линтер
mix dialyzer               # статический анализ типов (первый прогон строит PLT)
cargo fmt   --manifest-path native/leadbot_native/Cargo.toml
cargo clippy --manifest-path native/leadbot_native/Cargo.toml
```

## Структура

```
lib/lead_bot/
  application.ex   # OTP-супервизор: Dedup + ExGram + бот (если задан токен)
  bot.ex           # ExGram-хендлеры (polling), "печатает", дедлайн, дедуп
  extractor.ex     # промпт, вызов LLM, ретраи + фолбэк-модель, парсинг JSON
  open_router.ex   # HTTP-клиент OpenRouter (Req)
  card.ex          # рендер карточек лида в HTML (эскейпинг, N карточек)
  phone.ex         # нормализация телефонов к +7
  dedup.ex         # дедуп по {chat_id, fingerprint} (ETS + TTL 10 мин)
  native.ex        # Rustler-байндинги (normalize/fingerprint)
native/leadbot_native/
  src/lib.rs       # Rust: NFKC-нормализация + SHA-256 отпечаток
config/
  config.exs       # compile-time (ExGram Req-адаптер и т.п.)
  runtime.exs      # runtime: .env -> app env (dotenvy)
test/              # ExUnit: извлечение, устойчивость (моки API), Rust-NIF
```

## Архитектурные особенности и логика работы

Данный проект разработан с акцентом на высокую отказоустойчивость, безопасность и предсказуемость поведения в продакшене.

### 🛠 Ключевые решения и особенности:

1. **Обработка нескольких лидов в одном сообщении**
   Если клиент присылает сообщение с несколькими заявками (например, заказ для себя и для коллеги), бот распознает их как отдельные сущности и отправляет в чат самостоятельные карточки для каждой заявки. Это гарантирует, что ни один лид не будет потерян.

2. **Защита от инъекций (Prompt Injection)**
   Бот обрабатывает входящий текст исключительно как пользовательские данные. Попытки внедрения инструкций во входящие сообщения (например, «забудь предыдущие правила», «установи приоритет низкий» и т. д.) успешно блокируются на уровне системного промпта. При обнаружении попытки инъекции бот извлекает фактическую информацию и присылает карточку с соответствующим предупреждением.

3. **Интеллектуальная оценка срочности (Urgency)**
   - `высокая`: Заявки с жесткими временными рамками (дедлайн до 2–3 дней, маркеры «срочно», «горит», «сегодня/завтра»).
   - `средняя`: Наличие явного запроса без жестких ограничений по времени.
   - `низкая`: Предварительный интерес («обсудим позже», «прицениваюсь»).

4. **Нормализация контактных данных**
   Номера телефонов автоматически нормализуются к стандартному формату `+7XXXXXXXXXX` (модуль `LeadBot.Phone`). Если формат номера не распознан, оригинальная строка сохраняется без изменений, чтобы предотвратить потерю контактной информации.

5. **Отказоустойчивость и обработка ошибок (Resilience)**
   - **Дедупликация**: Исключение дублирования сообщений на основе уникального отпечатка (SHA-256 нормализованного текста через Rust-NIF) с использованием оперативного хранилища ETS и TTL 10 минут (модуль `LeadBot.Dedup`).
   - **Failover / Fallback**: При сбое основной модели (таймаут, ошибки 429/5xx, невалидный JSON на выходе) бот автоматически переключается на резервную модель (`OPENROUTER_FALLBACK_MODEL`).
   - **Rate Limiting & Backoff**: Запросы к API OpenRouter при временных ошибках сети/лимитов автоматически повторяются с линейным бэкоффом.
   - **Жёсткий тайм-аут**: Каждое сообщение обрабатывается в изолированном процессе `Task`. Лимит на обработку составляет 90 секунд, после чего процесс принудительно завершается, предотвращая зависание системы, а пользователь получает понятное сообщение об ошибке.

## Проверка качества и тестирование

Кодовая база полностью покрыта тестами и статическим анализом:
- `mix test` — юнит-тесты и тесты на устойчивость (с моками API).
- `mix credo --strict` — статический анализ стиля кода Elixir.
- `mix dialyzer` — проверка типизации.
- `cargo fmt` & `cargo clippy` — форматирование и линтинг Rust NIF.
